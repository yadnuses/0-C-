## 〇、开始之前：什么是"程序结构"？

前面几个模块你学会了写函数（M4）。但真实的程序往往很大，几百上千行。如果全塞进一个 `main`，那就成了一锅粥，没法读、没法改、没法和别人合作。

所以这一章讲的是**怎么把一个大程序"组织"起来**，主要四件事：

1. **函数互相调用**——一个函数里调用另一个函数（嵌套调用），甚至函数调用它自己（**递归**）。
2. **宏定义 `#define`**——在真正编译之前，先做一轮"查找替换"，给数字起名字、把短代码展开。
3. **多文件**——把程序拆成好几个 `.c` 文件和 `.h` 头文件，分工协作。
4. **变量存储类别**——`static`、`extern` 这些关键字，管的是"变量活多久、谁能看见它"。

类比：写一本书，你不会把所有字挤成一段。你会分章、分节、分段，还会做目录、写索引。程序结构干的就是这个"分章分节"的活。

## 一、函数嵌套调用：函数里再喊别的函数 ⭐

### 什么是嵌套调用？

**嵌套调用**说白了就是：A 函数干活干到一半，喊 B 函数来帮忙；B 干活干到一半，又喊 C 来帮忙。

C 语言里**任何函数都能调用别的函数**（除了不能在一个函数的身体里再"定义"另一个函数，但"调用"随便调）。`main` 调用 `cal`，`cal` 再调用 `vol_ball`，这就是嵌套。

```c
#include <stdio.h>

double square(double x) {       // 第3层：算平方
    return x * x;
}

double area(double r) {         // 第2层：算圆面积，要用到平方
    return 3.14159 * square(r); // area 调用了 square
}

int main(void) {                // 第1层：入口
    printf("%.2f\n", area(2.0));// main 调用了 area
    return 0;
}
// 输出：12.57
```

### 执行顺序：去的时候一层层进，回来的时候一层层退 ⭐

上面这段，电脑实际的跑法是这样的（务必看懂，这是理解递归的基础）：

```
main 开始
  → 调用 area(2.0)，main 先停在这等结果
      → area 里要算 square(2.0)，area 先停在这等结果
          → square(2.0) 算出 4.0，返回
      → area 拿到 4.0，算出 3.14159 * 4.0 = 12.57，返回
  → main 拿到 12.57，打印出来
main 结束
```

⚠️ 关键观念：**谁调用谁，就得等被调用的那个跑完、把结果交回来，自己才能继续。** 就像你让同事帮你查个数据，数据没回来之前你这边的活就卡着。这个"去的时候往里钻、回来的时候往外退"的过程，下一节的递归会反复用到。

### 结构化程序设计三原则（选择题爱考）📚

翁恺书里把"怎么拆函数"总结成三句话：

1. **自顶向下**——先想整体框架，再想细节。先画大盒子，再往里填小盒子。
2. **逐步求精**——把一个大步骤拆成几个小步骤，小步骤还嫌大就再拆。
3. **模块化（函数实现）**——把拆到最细的小目标，每个封装成一个函数。

设计函数时还有几条经验：
- 一个函数别太长（太长就该拆），方便读和调试。
- 重复的代码提取成一个公共函数，别复制粘贴。
- 少用全局变量，尽量用**参数传进去、返回值传出来**。

## 二、递归：函数自己调用自己 ⭐⚠️ 全章最重要

### 先讲个故事：什么叫递归

你排队买饭，看不见队头还有几个人。你问前面一个人："你前面几个人？" 他也看不见，于是他问他前面的人："你前面几个人？"……一直传到队头第一个人，他说"我前面 0 个"，然后答案一个一个往回传：1、2、3……最后传回到你。

这就是**递归**：**自己解决不了，就把问题甩给"更小一号的同一个问题"，直到小到能直接回答为止，再把答案一层层传回来。**

在 C 里，递归就是**一个函数在自己的身体里调用它自己**。

```c
int fact(int n) {
    if (n <= 1) return 1;        // ← 小到能直接回答了
    return n * fact(n - 1);      // ← 否则甩给更小一号的自己
}
```

看到没？`fact` 函数里又写了一句 `fact(n-1)`，**它调用了它自己**。这就是递归。

### 递归的两个必备要素 ⭐⚠️⚠️（必考，缺一不可）

写任何递归，脑子里必须有这两样东西，少一个程序就崩：

| 要素 | 别名 | 作用 | 缺了会怎样 |
|------|------|------|-----------|
| **① 终止条件** | base case / 递归出口 | 什么时候**停下来**，不再往下调自己 | 无限递归，栈溢出崩溃 |
| **② 递推关系** | 递归式子 | 怎么把**大问题拆成小一号的同类问题** | 根本没法用递归 |

> 🎯 真题口径：问"递归的两个关键点（必备条件）是什么？" 标准答案就是 **① 递归出口（终止条件）② 递归式子（把问题化为规模更小的同类子问题）**。背死它。

⚠️ 易错点：**终止条件必须写，而且必须保证每次递归都在"靠近"它。** 如果你写 `fact(n-1)` 但终止条件是 `n == 0`，而传进来的 n 是负数，那就永远到不了 0，无限往下调，最后**栈溢出（stack overflow）崩溃**。

### 阶乘 `n!` 逐步拆解执行过程 ⭐（看懂这个，递归就懂一半）

先复习一下阶乘：`n! = n × (n-1) × (n-2) × … × 2 × 1`。比如 `4! = 4×3×2×1 = 24`。

数学上它有个漂亮的规律：**`n! = n × (n-1)!`**。也就是"4 的阶乘 = 4 乘以 3 的阶乘"。这就是天生的递推关系。

```c
int fact(int n) {
    if (n <= 1) return 1;        // ① 终止条件：1! = 1，0! = 1，到这就别往下了
    return n * fact(n - 1);      // ② 递推关系：n! = n * (n-1)!
}
```

现在我们手动模拟 `fact(4)` 的全过程。**核心是看清"先一层层往里钻（展开），再一层层往外退（求值）"**：

**第一阶段：往里钻（每次都还差一个 fact 没算出来，只能挂起等待）**
```
fact(4) = 4 * fact(3)      ← 想知道答案，得先知道 fact(3)，挂起
  fact(3) = 3 * fact(2)    ← 想知道 fact(3)，得先知道 fact(2)，挂起
    fact(2) = 2 * fact(1)  ← 想知道 fact(2)，得先知道 fact(1)，挂起
      fact(1) = 1          ← 撞到终止条件！n<=1 直接返回 1，不再往下
```

**第二阶段：往外退（拿到底层答案后，一层层乘回去）**
```
      fact(1) = 1
    fact(2) = 2 * 1 = 2          ← fact(1) 回来了，算出 fact(2)
  fact(3) = 3 * 2 = 6            ← fact(2) 回来了，算出 fact(3)
fact(4) = 4 * 6 = 24            ← fact(3) 回来了，算出最终答案 24
```

⚠️ 一定要建立这个画面感：**递归不是"算一遍就完"，而是先一路俯冲到底（终止条件），再一路爬升返回。** 没到底之前，上面每一层都"卡着、等着"，这跟前面嵌套调用"调用方要等被调用方返回"是一模一样的道理，只不过这里调用方和被调用方是同一个函数。

### 递归调用栈：那些"挂起等待"的函数住在哪？⭐⚠️

你可能会问：`fact(4)` 还没算完就去算 `fact(3)`，那 `fact(4)` 里的 `n=4` 跑哪去了？会不会被 `fact(3)` 的 `n=3` 覆盖掉？

**不会。** 这就要讲到**调用栈（stack）**这个概念了：

- 每次调用一个函数（包括调用自己），系统都会在内存里开一块**新的小空间**（叫"栈帧"），专门放这一次调用的局部变量（比如这一层的 `n`）。
- 所以 `fact(4)`、`fact(3)`、`fact(2)`、`fact(1)` 各有**各自独立的 `n`**，互不干扰。它们像叠盘子一样一层层摞起来。
- 当某一层 `return` 了，它的那块空间就被弹走、回收，控制权交回上一层。

画个图（栈是从下往上摞、从上往下收的）：

```
往里钻时栈越摞越高：        到底后往外退，栈越收越矮：
┌──────────┐ fact(1) n=1    ← 这层先 return，弹走
├──────────┤ fact(2) n=2    ← 接着算完 return，弹走
├──────────┤ fact(3) n=3    ← 接着算完 return，弹走
├──────────┤ fact(4) n=4    ← 最后算完 return，弹走
└──────────┘ main
```

⚠️⚠️ **这就是"递归占栈空间"的真正含义**：递归多深，栈就摞多高。如果递归太深（比如几十万层）或者忘了写终止条件无限递归，栈空间被撑爆，程序就**栈溢出崩溃（Segmentation fault / stack overflow）**。这是递归相比循环最大的代价，常考。

### 📝 自己动手：逆序输出一个整数（翁恺例10-3）

题目：输入 12345，输出 54321。用递归怎么做？

- **递推关系**：先把**个位**（`num % 10`）输出，再把**剩下的高位**（`num / 10`）递归处理。
- **终止条件**：`num` 只剩一位数（`num <= 9`）时，直接输出它。

```c
void reverse(int num) {
    printf("%d", num % 10);          // 先输出个位
    if (num > 9)                     // 还有高位才继续
        reverse(num / 10);           // 递归处理剩下的高位
}
// reverse(123) 输出：321
```

模拟 `reverse(123)`：先打 `3`（123%10），再 `reverse(12)`→先打 `2`，再 `reverse(1)`→打 `1` 且 1≤9 停止。合起来就是 `321`。

## 三、汉诺塔：递归的"镇山之宝" ⭐⚠️🎯 几乎年年考

### 游戏规则

有三根柱子 A、B、C。A 上面套着 n 个盘子，**从下到大、从上到小**摞着（像一座塔）。目标：把这 n 个盘子整体搬到 C 上，规则两条：

1. **一次只能搬一个盘子**（搬最上面那个）。
2. **任何时候，大盘子都不能压在小盘子上面**。

B 可以当中转站。问：怎么搬？要搬多少次？

### 别想着"具体怎么搬"，要用递归的脑子想 ⭐⚠️

新手一上来就想"先搬哪个再搬哪个"，n 大一点就晕了。**递归的精髓是：不要纠结细节，只想"怎么把 n 个的问题，变成 n-1 个的问题"。**

假设你已经"会搬 n-1 个盘子"了（先信这个，这就是递归的信仰）。那么搬 n 个盘子只要三步：

```
① 把上面的 n-1 个盘子，从 A 借助 C，挪到 B          （调用自己，规模 n-1）
② 把最底下那个最大的盘子，从 A 直接挪到 C            （就一步，打印一行）
③ 把那 n-1 个盘子，从 B 借助 A，挪到 C              （调用自己，规模 n-1）
```

- **终止条件**：只剩 1 个盘子（`n == 1`）时，直接从源柱搬到目标柱，一步搞定，不用再拆。
- **递推关系**：上面的①和③，把"搬 n 个"化成了两次"搬 n-1 个"。

### 逐行讲解代码 ⭐

```c
#include <stdio.h>

// n: 要搬的盘子数; from: 源柱; via: 中转柱; to: 目标柱
void hanoi(int n, char from, char via, char to) {
    if (n == 1) {                              // ① 终止条件：只剩1个
        printf("%c -> %c\n", from, to);        //   直接从 from 搬到 to
        return;                                //   搬完立刻返回，别往下走
    }
    hanoi(n - 1, from, to, via);               // ② 把上面 n-1 个：A 借助 C 挪到 B
    printf("%c -> %c\n", from, to);            // ③ 最大那个：A 直接挪到 C
    hanoi(n - 1, via, from, to);               // ④ 把 n-1 个：B 借助 A 挪到 C
}

int main(void) {
    hanoi(3, 'A', 'B', 'C');                   // 把3个盘子从A搬到C，B中转
    return 0;
}
```

逐句拆：
- `void hanoi(int n, char from, char via, char to)` —— 四个参数。**最容易错的是搞混后三个柱子的顺序**。记法：`from`(从哪来) `via`(借哪过) `to`(到哪去)。
- `if (n == 1) { ... return; }` —— 终止条件。只剩一个盘子，直接搬，搬完 `return` 走人。⚠️ 这个 `return` 很重要，不写它会继续往下执行下面的递归，逻辑就乱了。
- 第②行 `hanoi(n-1, from, to, via)` —— 注意这里把原来的 `to`(C) 当成了**中转**，把原来的 `via`(B) 当成了**目标**。意思是"借助 C 把 n-1 个挪到 B"。
- 第③行 `printf` —— 把第 n 个（最大的）从 from 搬到 to。
- 第④行 `hanoi(n-1, via, from, to)` —— 现在从 `via`(B) 出发，借助 `from`(A)，搬到 `to`(C)。

`hanoi(3,'A','B','C')` 的输出（7 行）：
```
A -> C
A -> B
C -> B
A -> C
B -> A
B -> C
A -> C
```
数一数，正好 **7 次**。

### 移动次数公式：2ⁿ − 1 ⭐⚠️🎯（必考，背死）

搬 n 个盘子，最少需要移动的次数是：

> ## **次数 = 2ⁿ − 1**

为什么？因为搬 n 个 = 搬 n-1 个（一次）+ 搬最大那个（1 次）+ 再搬 n-1 个（一次）。设搬 n 个要 f(n) 次，则：
```
f(n) = 2·f(n-1) + 1
f(1) = 1
```
一路算：f(1)=1, f(2)=3, f(3)=7, f(4)=15, f(5)=31……正好是 **2ⁿ−1**。

| 盘子数 n | 移动次数 2ⁿ−1 |
|---------|--------------|
| 1 | 1 |
| 2 | 3 |
| 3 | 7 |
| 4 | 15 |
| 5 | 31 |
| 10 | 1023 |
| 64 | 18446744073709551615（一千八百多亿亿次，搬到地球毁灭也搬不完）|

🎯 真题常见两种问法：
1. "搬 n 个盘子需要移动几次？"→ 答 **2ⁿ−1**。
2. "搬 4 个盘子需移动几次？"→ 套公式 2⁴−1 = **15**。

## 四、递归 vs 循环：什么时候用哪个？ ⭐

很多能用递归写的，也能用循环写。比如阶乘：

```c
// 递归版：简洁，像数学公式
int fact_recur(int n) {
    if (n <= 1) return 1;
    return n * fact_recur(n - 1);
}

// 循环版：啰嗦一点，但不占栈
int fact_loop(int n) {
    int result = 1;
    for (int i = 2; i <= n; i++)
        result *= i;
    return result;
}
```

两者对比：

| | 递归 | 循环 |
|---|------|------|
| **代码** | 简洁、贴近数学定义、好理解 | 相对啰嗦 |
| **内存** | ⚠️ 每层调用都占栈空间，深了会**栈溢出** | 只用固定几个变量，省内存 |
| **速度** | 慢一点（函数调用有开销） | 快 |
| **适合** | 问题天生是"分成同类小问题"（汉诺塔、树、递归定义） | 简单重复（求和、累乘、遍历） |

⚠️ 结论：
- **能用简单循环干净解决的**（比如累加、累乘），优先用循环，省内存又快。
- **问题本身就是递归结构的**（汉诺塔、目录树、表达式解析），用递归代码会清爽得多，硬写循环反而费劲。

> 🎯 考研重点其实不是让你纠结"该用哪个"，而是：**给你一段递归代码，让你读懂它的执行过程、算出它的输出、或者算它被调用了多少次。** 所以前面"逐步拆解执行过程"那部分才是真正的考点，务必练熟。

## 五、斐波那契递归 & 它的"重复计算"毛病 ⭐⚠️

### 斐波那契数列

数列：`1, 1, 2, 3, 5, 8, 13, 21, ...`，规律是**每一项 = 前两项之和**。

数学定义天生就是递归的：
```
F(1) = 1
F(2) = 1
F(n) = F(n-1) + F(n-2)    (n > 2)
```

照着写成递归，毫不费力：

```c
int fib(int n) {
    if (n <= 2) return 1;            // 终止条件：第1、2项都是1
    return fib(n - 1) + fib(n - 2);  // 递推：前两项之和
}
```

### ⚠️⚠️ 致命问题：疯狂重复计算（高频考点）

斐波那契递归虽然好写，但**效率极低**，原因是**同一个值被反复算了无数遍**。看 `fib(5)` 的展开：

```
fib(5)
├─ fib(4)
│  ├─ fib(3)
│  │  ├─ fib(2)   = 1
│  │  └─ fib(1)   = 1
│  └─ fib(2)      = 1      ← fib(2) 又算了一遍
└─ fib(3)                  ← fib(3) 整个又算了一遍！
   ├─ fib(2)      = 1      ← fib(2) 第三遍
   └─ fib(1)      = 1
```

看出来了吗？`fib(3)` 被算了 2 次，`fib(2)` 被算了 3 次。n 越大，重复越离谱——`fib(50)` 要算几十亿次，电脑能卡到怀疑人生。

⚠️ 这是因为递归调用像一棵"二叉树"一样疯狂分叉，**左右两个分支算的东西大量重叠，谁也不知道对方已经算过了**。

🎯 真题常考：
- "斐波那契递归为什么效率低？"→ 因为**存在大量重复子问题的重复计算**。
- 让你画 `fib(n)` 的递归调用树，或数 `fib(n)` 一共被调用了多少次。

解决办法（了解即可）：用循环从前往后递推算一遍（每个只算一次），或者用数组把算过的存起来（叫"记忆化"）。考研知道"递归版有重复计算缺陷、循环版高效"就够了。

## 六、宏定义 #define：编译前的"查找替换" ⭐⚠️🎯 第二大考点

### 先搞懂"编译预处理"是个啥

你写的 `.c` 代码，在真正编译之前，会先经过一道"**预处理**"工序。预处理器专门处理以 `#` 开头的那些行（`#include`、`#define` 都是），**它干的活很简单粗暴：就是纯文本的查找、替换、粘贴。**

类比：你在 Word 里按 `Ctrl+H`，把全文所有的"小明"替换成"张三"。预处理器对 `#define` 干的就是这个，**纯文本替换，根本不管 C 语法、不做任何计算**。这句话是理解后面所有坑的钥匙。

### 不带参数的宏：给值起个名字 ⭐

```c
#define PI 3.14159
```

逐字拆：
- `#define` —— 预处理指令，"我要定义一个宏"。
- `PI` —— **宏名**，习惯**全大写**（为了和普通变量、函数区分开，一眼看出"这是个宏"）。
- `3.14159` —— 替换成的内容（叫"宏体"）。

作用：在编译前，把代码里**所有出现的 `PI`，原样替换成 `3.14159`**。

```c
#define PI 3.14159
double area = PI * r * r;
// 预处理后实际变成：
double area = 3.14159 * r * r;
```

好处：① 改一处全改（圆周率想更精确，只改 `#define` 那一行）；② 代码可读性好（看到 `PI` 就懂，比看到 `3.14159` 清楚）。

### ⚠️⚠️ 三个最容易扣分的细节

**坑 1：行尾绝对不要加分号！** 🎯

```c
#define PI 3.14159;        // ❌ 错！多了个分号
double area = PI * r * r;  // 替换后变成 3.14159; * r * r —— 语法错误！
```

宏是纯文本替换。你在 `#define` 后面加了分号，那个分号也会被**原封不动替换进去**，结果代码里冒出个莫名其妙的分号，报错。

> ⭐ 记死：**`#define` 行尾没有分号。** `#define PI 3.14159` 后面什么都不加，直接换行。

**坑 2：宏不是变量，没有类型，不占内存。** 它只在编译前存在，编译完就"消失"了（全被替换成了具体内容）。所以不能对宏取地址、不能 `scanf` 给宏赋值。

**坑 3：宏可以嵌套（宏里用宏）。**
```c
#define PI 3.1415926
#define AREA(r) (PI * (r) * (r))   // AREA 里用到了 PI，合法
```

### 带参数的宏：长得像函数，其实是"模板替换" ⭐⚠️

宏还能带参数，长得很像函数：

```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))   // 求两数较大者
#define SQR(x) ((x) * (x))                   // 求平方
```

调用 `MAX(3, 5)`，预处理器把它替换成 `((3) > (5) ? (3) : (5))`，算出 5。

⚠️ 但注意：它**不是函数调用**！没有"传参、计算、返回"那一套，**纯粹是把 `MAX(3,5)` 这串文字，按模板展开成另一串文字**。

### 🎯🎯 带参宏的经典大坑：括号！（几乎必考）

这是整个第10章最爱考的点，给我打起十二分精神。

**坑 A：参数不加括号，遇到表达式实参就出错**

看这个"看起来没毛病"的平方宏：

```c
#define SQR(x) x*x          // ❌ 危险写法：x 两边没括号
```

调用 `SQR(2+3)`，你以为是 `(2+3)² = 25`。但预处理器只会**死板地把 `x` 替换成 `2+3`**：

```c
SQR(2+3)  →  2+3*2+3
```

而 `2+3*2+3` 按运算优先级（先乘后加）算 = `2 + 6 + 3` = **11**，根本不是 25！

⚠️ 原因：宏是纯文本替换，**它不会帮你把实参 `2+3` 当成一个整体**，直接怼进去，于是 `*` 抢在 `+` 前面算了。

**正确写法：每个参数都用括号包起来，整个宏体也用括号包起来。**

```c
#define SQR(x) ((x)*(x))    // ✅ 正确：每个 x 加括号，外面再加一层
```

这下 `SQR(2+3)` 展开成：
```c
((2+3)*(2+3))  →  (5*5)  →  25   ✓
```

**坑 B：外层不加括号，宏整体被外面的运算"撕开"**

```c
#define DOUBLE(x) (x)+(x)     // 内层加了括号，但整体没加
int y = DOUBLE(3) * 2;        // 你以为是 (3+3)*2 = 12
// 展开：(3)+(3) * 2  →  3 + 6  →  9   ✗ 不是 12！
```

因为外面那个 `* 2` 把 `(3)+(3)` 撕开了，只跟最后一个 `(3)` 结合。

**所以带参宏的黄金法则 ⭐⚠️🎯（背下来，照着写绝不出错）：**

> ## 每个参数都加括号，整个宏体外面再套一层括号。
> `#define SQR(x) ((x)*(x))`

```c
#define SQR(x) ((x)*(x))                  // ✅
#define MAX(a,b) ((a)>(b)?(a):(b))        // ✅ a、b 各自加括号，外面套括号
#define ADD(a,b) ((a)+(b))                // ✅
```

### 宏 vs 函数 区别对照表 ⭐（选择题常考）

| 对比项 | 带参宏 `#define` | 函数 |
|--------|-----------------|------|
| 处理时机 | **编译预处理阶段**（编译前） | **程序运行时** |
| 本质 | 纯文本替换，不计算参数 | 先算实参的值，再传进去 |
| 参数类型 | 没有类型概念（什么都能塞） | 有明确类型 |
| 占内存/时间 | 不调用、无开销，但代码会膨胀（每处都展开一遍） | 有调用开销（压栈、跳转），但代码只一份 |
| 速度 | 快（没调用开销） | 略慢（有调用开销） |
| 副作用风险 | ⚠️ 大（`SQR(i++)` 会让 i 自增两次！） | 小 |

⚠️ 补一个副作用例子加深印象：`SQR(i++)` 展开成 `((i++)*(i++))`，`i` 被自增了**两次**，结果完全不是你想要的。函数 `sqr(i++)` 就只自增一次。这也是"宏不如函数安全"的体现。

## 六点五、⭐⭐ 考研要求的约定：真题里那套 #define 和 typedef（必看）

> 带参宏的括号坑讲完了。但你刷**王道、408 大题**时，会发现每段数据结构代码**开头都顶着几行固定的 `#define` 和 `typedef`**，长得很统一。它们用的全是你刚学的宏定义和 M5/M8 的类型知识，但**约定俗成的样子**得提前认脸，否则读真题代码时第一行就卡住。

### ① `#define MAXSIZE 100`：用宏定数组容量 ⭐

真题的顺序表、栈、队列，容量几乎都用宏写死：

```c
#define MAXSIZE 100        // 顺序表最多放 100 个元素

typedef struct {
    ElemType data[MAXSIZE]; // ← 直接用宏 MAXSIZE 当数组长度
    int length;
} SqList;
```

- 这就是你刚学的"不带参宏"——编译前把所有 `MAXSIZE` 替换成 `100`。
- 好处：想改容量只改这一行。⚠️ 还是那条铁律：**`#define MAXSIZE 100` 行尾不加分号**。
- ⚠️ 数组长度用 `MAXSIZE`（宏，编译时就是常量），这正是 M6 说的"数组长度必须是确定的常量"——宏替换后就是个常量，合法。

### ② `OK / ERROR / Status`：真题的"返回状态"约定 ⭐⚠️

408 教材（严蔚敏那套）的算法函数，返回值不直接用 `int`，而是这么一套：

```c
#define OK    1            // 操作成功
#define ERROR 0            // 操作失败
typedef int Status;        // 给 int 起别名叫 Status，专门当"状态码"用
```

于是真题里的插入、删除函数长这样：

```c
Status ListInsert(SqList *L, int i, ElemType e) {
    if (i < 1 || i > L->length + 1) return ERROR;   // 不合法，返回 ERROR(就是0)
    // ……插入……
    return OK;                                       // 成功，返回 OK(就是1)
}
```

⚠️ 拆穿这套"方言"，全靠你已学的知识：
- `Status` 不是新类型，就是 **`int` 起的别名**（M8 的 typedef）。`Status ListInsert(...)` = `int ListInsert(...)`。
- `OK`、`ERROR` 不是关键字，就是**宏**（M9 这节），分别替换成 `1` 和 `0`。
- 所以 `return OK;` 本质就是 `return 1;`，`return ERROR;` 就是 `return 0;`。看到别懵。

> 📌 一句话：**`Status` = int 别名，`OK` = 1，`ERROR` = 0。真题函数 `Status f(...){... return OK;}` 翻译成你熟悉的就是 `int f(...){... return 1;}`。**

### ③ `bool / true / false`：C 也能用，但要加头文件 ⭐⚠️

有些真题代码用 `bool` 当返回类型、用 `true`/`false`：

```c
bool ListInsert(SqList &L, int i, ElemType e) {
    if (i < 1 || i > L.length + 1) return false;
    // ……
    return true;
}
```

⚠️ 关键辨析（呼应 M2「C 里 0 是假、非 0 是真」、M5「C 没有专门的真假类型」）：
- **纯老 C（C89）根本没有 `bool` 这个类型**，真假全靠 `int` 的 0 和非 0 表示。
- C99 起，**加 `#include <stdbool.h>` 才能用 `bool`、`true`、`false`**。本质上 `true` 就是 1、`false` 就是 0，`bool` 就是能装 0/1 的类型。
- 如果真题代码用了 `bool` 却没 `#include <stdbool.h>`，拿 `gcc` 按 C89 编译会报错——这时把 `bool/true/false` 换成 `int/1/0` 一样能跑。

> 📌 记住：**`bool/true/false` 是 C99 的东西，要 `#include <stdbool.h>`；它和 `Status`+`OK/ERROR` 是两套等价的"状态表示法"，底层都是 int 的 1 和 0。**

### ④ 完整对照：真题函数头 vs 你学的纯 C

```c
// 【真题/王道写法】顶着一堆约定
#define MAXSIZE 100
#define OK 1
#define ERROR 0
typedef int Status;
typedef int ElemType;

Status ListInsert(SqList *L, int i, ElemType e) { ... return OK; }

// 【翻译成你最熟悉的纯 C】
//   Status → int，ElemType → int，OK → 1，ERROR → 0，MAXSIZE → 100
int ListInsert(SqList *L, int i, int e) { ... return 1; }
```

两段**完全一样**，只是真题那套用宏和 typedef 包装了一层，显得"专业、好改"。**你以后写大题也照真题这套来，阅卷老师看着顺眼。** 现在能"看穿这层包装"就够了。

## 七、文件包含与多文件编译 ⭐

### 为什么要拆成多个文件？

一个上千行的程序全堆在一个 `.c` 里，没法看、没法分工。所以大程序拆成多个文件，每个文件管一块功能。这就涉及两类文件：

| 文件类型 | 放什么 | 类比 |
|---------|--------|------|
| **`.h` 头文件** | **声明**：函数原型、结构体定义、宏定义、`extern` 变量声明 | 一本书的"目录"——告诉你有啥，但不展开内容 |
| **`.c` 源文件** | **定义**：函数的具体实现代码 | 书的"正文"——真正的内容在这 |

> ⭐ 一句话记牢：**`.h` 放声明（告诉别人"有这个东西"），`.c` 放定义（真正实现它）。**

### `#include` 的两种写法：尖括号 vs 双引号 ⭐⚠️

```c
#include <stdio.h>      // 尖括号：到【系统标准目录】找（系统自带的库）
#include "myhead.h"     // 双引号：先到【当前目录】找，找不到再去系统目录
```

`#include` 的作用：**把那个文件的全部内容，原封不动地"复制粘贴"到 `#include` 这一行的位置。** 它也是预处理指令，干的还是"文本粘贴"的活。

记法：
- `< >` 尖括号 —— 标准库（`stdio.h`、`math.h`、`string.h` 这些**别人写好的、系统自带的**），直接去系统目录找。
- `" "` 双引号 —— **你自己写的头文件**，先在你项目的当前目录找。

> ⭐ 考点：引自己写的头文件用**双引号** `#include "myhead.h"`；引系统库用**尖括号** `#include <stdio.h>`。

### 一个完整的多文件例子

假设把"加法函数"单独拆出去。三个文件：

**`add.h`（头文件，放声明）**
```c
int add(int a, int b);     // 只声明：有这么个函数，长这样。注意有分号
```

**`add.c`（源文件，放定义）**
```c
#include "add.h"           // 把自己的声明也包含进来，保证一致
int add(int a, int b) {    // 真正的实现
    return a + b;
}
```

**`main.c`（主文件，用它）**
```c
#include <stdio.h>
#include "add.h"           // 包含进来，就能用 add 了
int main(void) {
    printf("%d\n", add(3, 5));   // 输出 8
    return 0;
}
```

### 怎么编译多个文件？⭐

把所有 `.c` 文件**一起**丢给编译器（注意：`.h` 不用写在命令里，它会通过 `#include` 自动被带进去）：

```bash
gcc main.c add.c -o program     # 两个 .c 一起编译，生成可执行文件 program
./program                          # 运行
```

⚠️ 新手常犯：只编译 `main.c`（`gcc main.c`），结果报错 `undefined symbol: add`——因为 `add` 的实现在 `add.c` 里，你没把它一起编译，链接器找不到 `add` 的身体。**记住：用到几个 `.c`，编译命令里就都得列上。**

### #ifndef / #define / #endif：防止头文件被重复包含（了解）⭐

问题：如果 `a.h` 和 `b.h` 都 `#include "common.h"`，而你的 `main.c` 又同时包含了 `a.h` 和 `b.h`，那 `common.h` 的内容就被粘贴了**两遍**，里面的结构体、定义重复出现，编译报"重复定义"。

解决：给每个头文件套一层"**防护罩**"，叫**头文件保护（include guard）**：

```c
#ifndef COMMON_H        // 如果还没定义过 COMMON_H 这个标记
#define COMMON_H        // 就定义它（插个旗，表示"我进来过了"）

// ……头文件的真正内容（结构体、声明、宏）写在这中间……

#endif                  // 结束
```

工作原理：
- 第一次被包含：`COMMON_H` 还没定义，`#ifndef` 成立，于是定义 `COMMON_H` 并把内容编译进去。
- 第二次再被包含：`COMMON_H` 已经定义过了，`#ifndef` 不成立，中间的内容直接**整段跳过**，于是不会重复。

⚠️ 考研一般只要求**认识**这三行、知道**它是用来防止头文件重复包含的**就行，不用默写。`COMMON_H` 这个名字随便起（通常用文件名大写）。

## 八、条件编译：让代码"选择性"参与编译（了解）⭐

条件编译也是预处理的一种，长这样：

```c
#if 条件
    程序段1
#else
    程序段2
#endif
```

作用：**在编译之前**，根据条件决定"哪段代码参与编译、哪段干脆被删掉不编译"。常用于"调试版/正式版"切换、跨平台兼容。

```c
#define DEBUG 1

#if DEBUG
    printf("调试信息：x = %d\n", x);   // DEBUG为1，这行会被编译进去
#else
    // DEBUG为0时，上面那行根本不会出现在程序里
#endif
```

### ⚠️ 条件编译 `#if` vs 运行时 `if` 的区别（易混考点）⭐

| | 条件编译 `#if` | 普通 `if` 语句 |
|---|--------------|--------------|
| 什么时候判断 | **编译前**（预处理阶段） | **程序运行时** |
| 判断结果 | 决定**某段代码要不要编译进程序** | 决定运行时**走哪条分支** |
| 没选中的代码 | 根本不进最终程序（不占空间） | 仍在程序里，只是这次没执行 |

一句话：`#if` 是"编译时就把不要的代码删掉"，`if` 是"代码都在，运行时再挑一条走"。

## 九、变量的存储类别：变量能活多久、谁看得见 ⭐⚠️

到现在你写的变量，函数一结束就被销毁了。但有些场景需要变量"活得久一点"或"跨文件共享"，这就要用到**存储类别**关键字。考研主要考 `static` 和 `extern`。

### static 局部变量：函数结束了，值还留着 ⭐⚠️🎯

普通局部变量：函数每次调用都**重新出生**，函数一结束就**销毁**，下次进来又从头来。

`static` 局部变量：**只在第一次调用时出生、初始化一次；函数结束后它不销毁，值一直留着，下次进来接着用。**

```c
#include <stdio.h>

void counter(void) {
    static int count = 0;    // static：只初始化一次，函数结束值不丢
    count++;
    printf("第 %d 次调用\n", count);
}

int main(void) {
    counter();    // 第 1 次调用
    counter();    // 第 2 次调用
    counter();    // 第 3 次调用
    return 0;
}
```

输出：
```
第 1 次调用
第 2 次调用
第 3 次调用
```

⚠️ 对比：如果把 `static` 去掉，写成普通的 `int count = 0;`，那每次进来 `count` 都重新变回 0，自增成 1，输出就全是"第 1 次调用"。

`static` 局部变量的几个考点 ⭐：
- **初始化只执行一次**（第一次调用时），后续调用跳过初始化，沿用上次的值。
- 没赋初值的话，系统**自动赋 0**（普通局部变量不赋初值是随机垃圾值，这点不同）。
- **生存周期**：整个程序运行期间都活着（存在"静态存储区"，跟全局变量一块）。
- **作用域**：仍然只限于它所在的那个函数内部，别的函数看不见它。

> ⭐ 记法：`static` 局部变量 = "**生存周期长（一直活着）+ 作用域窄（只本函数能用）**"。用途：需要"记住上次结果"的场合，比如计数器、累乘器。

### extern：声明"这个变量定义在别的文件里" ⭐

多文件时，A 文件定义了一个全局变量 `Count`，B 文件想用它，怎么办？在 B 文件里用 `extern` **声明**一下：

```c
// file1.c —— 定义（真正分配内存就这一次）
int Count = 0;

// file2.c —— 声明（告诉编译器：Count 在别处定义了，我借用一下）
extern int Count;        // 注意：extern 声明不分配内存，不赋初值
void show(void) {
    printf("%d\n", Count);   // 就能用 file1.c 里的 Count 了
}
```

⚠️ 关键区分：
- **定义** `int Count = 0;` —— 真正造出这个变量、分配内存，**整个程序只能有一次**。
- **声明** `extern int Count;` —— 只是"打个招呼，说这名字在别处定义了"，**可以在多个文件里声明多次**，不分配内存。

> ⭐ 一句话：**全局变量只能定义一次，但可以用 `extern` 在多个文件里声明、借用。**

### 顺带认识：static 全局变量 / static 函数（了解）

`static` 用在**全局变量或函数**前面，意思和上面局部变量不同——它表示"**只限本文件用，别的文件不许访问**"：

- `static int x;`（写在函数外）—— 这个全局变量只有**本 `.c` 文件**能用，别的文件即使 `extern` 也访问不到。用来避免和别人的同名全局变量打架。
- `static void helper() {...}` —— 这个函数只能被**本文件**调用，别的文件调不了。增强封装。

补充：默认情况下，函数都是"外部的"，别的文件加个声明就能调用（`extern` 可省略）。加了 `static` 才会被"关在本文件里"。

## 十、报错了别慌：本章高频报错排查 ⚠️

| 报错现象 | 八成是这个原因 |
|---------|--------------|
| 程序运行直接崩溃 `Segmentation fault` | 递归忘了写终止条件，或终止条件永远到不了 → **栈溢出** |
| `#define` 那行附近语法错误 | `#define` 行尾**多写了分号** |
| 宏算出来的结果离谱（不是预期值） | 带参宏**漏了括号**，被运算优先级坑了 |
| 链接报错 `undefined symbol: xxx` | 多文件编译时**漏了某个 `.c`** 没一起编译 |
| `redefinition` / 重复定义 | 头文件被重复包含 → 加 `#ifndef/#define/#endif` 保护 |

万能口诀：
> **递归崩 → 查终止条件；宏出错 → 查括号和分号；链接错 → 查 .c 是否全编译。**

## 十一、本节考点速记 ⭐

1. **嵌套调用**：A 调 B，A 必须等 B 返回才继续；执行是"往里一层层钻，回来一层层退"。
2. **结构化设计三原则**：自顶向下、逐步求精、模块化（函数实现）。
3. **递归两要素（必考）**：① 终止条件（递归出口）② 递推关系（化为规模更小的同类子问题），缺一不可。
4. **递归执行**：先一路俯冲到终止条件，再一路返回求值；要会手动模拟、算输出。
5. **递归调用栈**：每层调用各有独立的局部变量；递归太深或无终止条件 → **栈溢出崩溃**。这就是"递归占栈空间"。
6. **汉诺塔**：三步法（n-1 个挪走 → 搬最大的 → n-1 个挪回）；终止条件 `n==1`。
7. **汉诺塔移动次数 = 2ⁿ − 1**（必背！4 个盘子 = 15 次）。
8. **递归 vs 循环**：递归简洁但占栈、慢；循环省内存、快。简单重复用循环，天然递归结构用递归。
9. **斐波那契递归的毛病**：大量**重复计算**同一子问题，效率极低（指数级）。
10. **`#define` 行尾绝不加分号**！宏是预处理阶段的**纯文本替换**，不计算、不管类型。
11. **带参宏黄金法则**：每个参数加括号、整个宏体外面再套括号 → `#define SQR(x) ((x)*(x))`。
12. **带参宏经典坑**：`#define SQR(x) x*x` 调用 `SQR(2+3)` 展开成 `2+3*2+3=11`，不是 25。
13. **宏 vs 函数**：宏在编译预处理阶段文本替换、不计算参数；函数在运行时调用、先算实参。
14. **`#include`**：尖括号 `<>` 找系统库；双引号 `" "` 引自己的头文件（先找当前目录）。
15. **多文件**：`.h` 放声明，`.c` 放定义；编译时所有 `.c` 一起编译（`gcc main.c add.c -o ...`）。
16. **`#ifndef/#define/#endif`**：防止头文件被重复包含。
17. **条件编译 `#if`**：编译前决定代码要不要编译进去；区别于运行时的 `if`。
18. **`static` 局部变量**：只初始化一次，函数结束值还留着，生存周期=整个程序，作用域=本函数；未赋初值自动为 0。
19. **`extern`**：声明变量定义在别的文件；全局变量只能定义一次，可多处 `extern` 声明。
20. **`static` 全局变量/函数**：限制作用域在本文件，别的文件访问不到。
